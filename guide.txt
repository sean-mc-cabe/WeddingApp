DAY 4 â€” POWER AUTOMATE ACTIONS, DATAVERSE WRITES, CARDS (2â€“3 h)
--------------------------------------------------------------------------------

Focus: make the agent do things, not just answer.

Build steps
 Create the flows INSIDE the LearnBot solution. Every flow uses connection
 references (not raw connections) â€” this matters for ALM on Day 7.

 1. Flow: flow_FindSession
      Trigger: When Copilot Studio calls a flow.
      Inputs: courseName (text), deliveryMode (text), preferredDate (text).
      Actions: List rows (Dataverse, lfi_session) filtered on course name +
      delivery mode + start date >= preferredDate, order by start date,
      top 1. Condition: any rows? Seats remaining > 0?
      Outputs: sessionCode (text), sessionStart (text), seatsRemaining (int),
      status (text: Found / NotFound / Full).
 2. Flow: flow_RegisterForSession
      Inputs: sessionCode, userUpn.
      Actions: Get session row, decrement seatsRemaining (if > 0), add row
      to lfi_registration (status Confirmed), return registrationCode and
      sessionStart. Handle race: if seatsRemaining already 0, return
      WaitlistedOrFailed.
 3. Flow: flow_CancelRegistration
      Inputs: registrationCode OR (userUpn + sessionCode).
      Actions: lookup registration, set status Cancelled, increment session
      seatsRemaining. Return confirmation text.
 4. Flow: flow_SubmitTopicRequest
      Inputs: description, businessJustification, userUpn.
      Actions: Add row to lfi_topicrequest (status New), post Adaptive Card
      summary to L&D Teams channel (using lfi_LnDTeamsChannelId env var),
      send confirmation email to submitter using lfi_LnDCoordinatorEmail.
 5. Flow: flow_GetSeatsRemaining
      Inputs: courseName (text), preferredDate (text).
      Actions: List rows, return the next 3 upcoming sessions with seat
      counts as a JSON array. Used by a deterministic topic (no LLM) so the
      answer is exact.

 In the agent:
 6. Add each flow as an Action. For each, write a crisp description â€” this
    becomes critical on Day 5 for generative orchestration.
 7. Wire flow_FindSession into "Register for a course" where the stub was.
    Branch on status. Wire flow_RegisterForSession on confirmation. Surface
    the registration code and session start time in a final Adaptive Card.
 8. Wire flow_CancelRegistration into the Cancel topic.
 9. Wire flow_SubmitTopicRequest into the Request topic.
10. Create a dedicated topic "How many seats are left" that does NOT use
    generative answers â€” it asks for course + date, calls
    flow_GetSeatsRemaining, and formats the JSON into a card. This is your
    "correctness matters" pattern.
11. Add global error handling: on action failure in any topic, set
    lastErrorMessage, message the user with a friendly apology plus the
    conversation ID (System.Conversation.Id) for support, and end gracefully.

Expected result
 Full end-to-end: user registers for a session, a Dataverse row lands,
 seatsRemaining decrements, a registration code comes back. Cancellation
 works. Topic request posts to Teams and emails the submitter. Seats
 query returns exact numbers with zero LLM involvement.

Break-it tasks
 - Disable the Teams action target or break the channel ID â€” does error
   handling actually fire, or does a raw error bubble?
 - Two browser tabs, simultaneous registration on a 1-seat session â€” does
   one succeed and one get a clean failure?
 - Submit a topic request with a 3,000-character description â€” Dataverse
   truncation? Card rendering?
 - Disconnect network mid-flow.

Key insights
 - The agent is a thin coordinator. Business logic lives in Power Automate
   and Dataverse. Treat agent as UI, flows as API.
 - Action timeouts are real and not configurable upward indefinitely. Long
   operations must go async ("we'll message you when done").
 - There is no transaction across multiple actions. Build idempotency and
   reconciliation. A failed Teams post after a successful Dataverse write
   leaves an orphan.
 - Connection references (not raw connections) are non-negotiable for ALM.

SLIDE 2 â€” What is Copilot Studio (and what it isn't)
--------------------------------------------------------------------------------

 - Low-code platform for building conversational + generative agents
 - Successor to Power Virtual Agents â€” same runtime, very different posture
 - NOT Microsoft 365 Copilot
 - NOT an agent in M365 Copilot (though it can power one)
 - NOT a replacement for Power Apps / Automate â€” it sits on top

Talking points
