# Solution Overview

## What This Solution Does

The FAQ Agent Factory v2 is a complete system that lets business units self-serve the creation and maintenance of FAQ content for Copilot Studio agents, while giving agent engineers a centralised management and monitoring layer. It eliminates the manual handoff between BUs and the engineering team for routine content updates.

## How It Works

- **Submit: **A BU user submits a policy document via a SharePoint form with required metadata (category, approver, target agent).

- **Extract: **A Power Automate flow extracts the document’s content as structured FAQ pairs using AI Builder.

- **Review: **The submitter receives an Adaptive Card in Teams showing the extracted FAQs. They edit, approve, or reject directly in the card.

- **Publish: **On approval, a flow generates a formatted FAQ document from a Word template and saves it to a SharePoint folder configured as the agent’s knowledge source. Copilot Studio automatically re-indexes.

- **Test: **The save triggers a second flow that generates test cases, stores them in Dataverse, and runs them against the agent via Direct Line API.

- **Self-serve: **BU users manage their FAQ content through a lightweight Canvas app — uploading new documents, editing individual FAQs, or removing content. Security groups control who sees what.

- **Monitor: **Agent engineers use a model-driven management app to oversee all agents, request BU reviews, run test suites, and track quality over time.

|  | **Key Architecture Change from v1** The knowledge source is now a SharePoint folder per agent, not uploaded file groups. Generated FAQ documents are saved directly to SharePoint. Copilot Studio reads from SharePoint via the connector and re-indexes automatically. This eliminates the manual knowledge file upload step entirely. |
| --- | --- |

# Data Model

Five Dataverse tables. The addition of a Source Documents table tracks the original uploaded files and links them to their extracted FAQs.

## Table 1: Agents (cr_agent)

| **Column** | **Type** | **Description** |
| --- | --- | --- |
| AgentName | Text (primary) | Display name of the agent |
| Description | Multiline text | What the agent covers |
| OwningBU | Text | Business unit name |
| BUSecurityGroup | Text | Entra security group ID for access control |
| BotId | Text | Copilot Studio bot ID |
| DirectLineSecret | Text | For test runner API calls |
| KnowledgeFolderURL | URL | SharePoint folder configured as knowledge source |
| Status | Choice | Active, Draft, Retired |
| LastPublished | Date/Time | When last FAQ doc was saved to knowledge folder |
| TestPassRate | Decimal | Current pass rate (0–1) |
| TotalFAQs | Whole number | Count of active FAQs |
| AgentInstructions | Multiline text | The agent’s system instructions (for reference) |

## Table 2: Source Documents (cr_sourcedocument)

Tracks every policy document submitted by BU users. Links to the FAQs extracted from it.

| **Column** | **Type** | **Description** |
| --- | --- | --- |
| DocumentName | Text (primary) | Name of the uploaded file |
| SharePointURL | URL | Link to the file in SharePoint |
| Agent | Lookup → Agents | Target agent |
| Category | Choice | HR, IT, Finance, Legal Ops, Facilities, Other |
| SubmittedBy | Text | Email of the submitter |
| SubmittedDate | Date/Time | When the document was submitted |
| Status | Choice | Processing, Extracted, Approved, Error, Archived |
| FAQDocURL | URL | Link to the generated FAQ document in the knowledge folder |
| FAQCount | Whole number | Number of FAQs extracted |

## Table 3: FAQ Items (cr_faqitem)

| **Column** | **Type** | **Description** |
| --- | --- | --- |
| Title | Text (primary, 500 chars) | The FAQ question |
| Answer | Multiline text | The approved answer |
| SourceDocument | Lookup → Source Documents | Which document this FAQ came from |
| Agent | Lookup → Agents | Which agent this FAQ belongs to |
| Category | Choice | HR, IT, Finance, Legal Ops, Facilities, Other |
| Status | Choice | Pending Review, Approved, Rejected, Needs Update, Archived |
| ApprovedBy | Text | Approver’s email |
| ApprovedDate | Date | Last approval date |
| NextReviewDate | Date | Auto: ApprovedDate + 90 days |
| Active | Yes/No | Controls inclusion in generated FAQ doc |
| SortOrder | Whole number | Order within generated document |
| SourceSection | Text | Section reference from original document |

## Table 4: Test Cases (cr_testcase)

| **Column** | **Type** | **Description** |
| --- | --- | --- |
| TestQuestion | Text (primary, 500 chars) | Paraphrased test question |
| ExpectedAnswer | Multiline text | Expected answer (from linked FAQ) |
| LinkedFAQ | Lookup → FAQ Items | The FAQ this test validates |
| Agent | Lookup → Agents | Agent to test against |
| LastResult | Choice | Pass, Fail, Not Run |
| LastRunDate | Date/Time | When last executed |
| ActualResponse | Multiline text | Agent’s actual response |
| SimilarityScore | Decimal | AI-judged similarity (0–1) |
| VariationType | Text | casual, scenario, partial, negative, followup |

## Table 5: Test Runs (cr_testrun)

| **Column** | **Type** | **Description** |
| --- | --- | --- |
| RunName | Text (primary) | Auto: AgentName + DateTime |
| Agent | Lookup → Agents | Which agent was tested |
| RunDate | Date/Time | When the run started |
| TotalTests | Whole number | Test cases executed |
| Passed | Whole number | Pass count |
| Failed | Whole number | Fail count |
| PassRate | Decimal | Passed / TotalTests |
| Trigger | Choice | Manual, Post-Publish, Scheduled |

# Implementation Guide

## Phase 1: Foundation

### Step 1.1: Create the Solution

- Go to make.powerapps.com, select your environment.

- Solutions → New solution. Name: “FAQ Agent Factory”.

- Set your publisher prefix (e.g. cr). All components will use this prefix.

### Step 1.2: Create All Five Dataverse Tables

Create each table inside the solution using the schema in Section 2. Build in this order to handle lookups correctly: Agents first, then Source Documents, then FAQ Items, then Test Cases, then Test Runs.

|  | **Enable Dataverse Search** Go to Power Platform Admin Centre → Environment → Settings → Features → Dataverse search = On. Required for knowledge source functionality. |
| --- | --- |

### Step 1.3: Create the SharePoint Site and Libraries

- Create a SharePoint site: “FAQ Agent Factory”.

- **Submissions library: **Add columns: Category (choice), Approver (person), Target Agent (choice — populate with agent names), Processing Status (choice: New, Processing, Extracted, Approved, Error).

- **Knowledge library: **Create one subfolder per agent (e.g. HR-Policy-Agent, IT-Policy-Agent).

- **Templates library: **Upload the FAQ document Word template (see Phase 3 for template design).

### Step 1.4: Create Security Groups

In Microsoft Entra (Azure AD), create a security group per agent/BU scope. Example: “FAQ-HR-Contributors”, “FAQ-IT-Contributors”. These groups control who can see and edit FAQ content in the BU Self-Service App. Record each group’s Object ID in the Agents table’s BUSecurityGroup column.

## Phase 2: Flow 1 — Document Submission & FAQ Extraction

**Trigger: **“When an item is created” — SharePoint connector, targeting the Submissions library.

### Step 2.1: Get File Content and Extract Text

- **Get file content: **Use SharePoint “Get file content” with the file identifier from the trigger.

- **Extract text: **Use AI Builder “Extract information from documents” or “Read text from document”. For PDFs this handles OCR. For Word files, text extraction.

- **Update status: **Update the SharePoint item’s Processing Status to “Processing”.

### Step 2.2: Generate FAQs with AI Builder

- **AI Builder GPT action: **Use “Create text with GPT using a prompt”. Use the FAQ Extraction prompt (see Section 4: AI Prompts). Map extracted text to the {{DocumentContent}} input variable.

- **Clean JSON: **Compose action to strip code fences: replace(replace(output,'```json',''),'```','')

- **Parse JSON: **Parse the cleaned output into typed objects.

### Step 2.3: Create Dataverse Records

- **Create Source Document row: **DocumentName = file name, SharePointURL = file URL, Agent = target agent from metadata, Category = from metadata, SubmittedBy = trigger user, Status = Extracted.

- **Loop through FAQs: **For each FAQ in the parsed array, create a FAQ Item row: Title = question, Answer = answer, SourceDocument = the row just created, Agent = target agent, Status = Pending Review, SourceSection = source_section from JSON.

### Step 2.4: Send Adaptive Card for Review

Use “Post adaptive card and wait for a response” (Teams connector). Send to the submitter (from trigger metadata).

The card design depends on FAQ count. For 15 or fewer FAQs, display them all in the card with individual approve/edit fields. For more than 15, show a summary with the first 5 FAQs and a link to the BU Self-Service App for full review.

**Card structure (summary version):**

- Document name, category, FAQ count

- Sample FAQs (first 5) showing question and answer

- Remaining count indicator

- Three action buttons: “Approve All”, “Review in App”, “Reject All”

**Card structure (full review version, ≤15 FAQs):**

- Each FAQ shown with the question as a header

- Answer in an editable Input.Text field (allows inline edits)

- Toggle per FAQ: Approve / Reject

- Submit button that sends all decisions back

|  | **Adaptive Card Size Limit** Teams enforces a 28KB limit on Adaptive Card payloads. With long FAQ answers, you can hit this with 8–10 FAQs. Test your card payload size and switch to the summary version if needed. The full review experience lives in the BU Self-Service App. |
| --- | --- |

### Step 2.5: Process the Response

Based on the card response action:

- **Approve All: **Update all FAQ Items for this document: Status = Approved, ApprovedBy = responder, ApprovedDate = utcNow(), NextReviewDate = addDays(utcNow(), 90), Active = Yes. Update Source Document Status = Approved. Then trigger Flow 2 (call as child flow or use a Dataverse trigger).

- **Review in App: **Leave FAQ status as Pending Review. Send a follow-up message with the app deep link filtered to this document’s FAQs.

- **Reject All: **Update all FAQ Items: Status = Rejected. Update Source Document Status = Error. Notify submitter.

- **Individual (full review card): **Parse the response data to get per-FAQ decisions. Update each FAQ Item accordingly. If any answers were edited in the card, update the Answer field. Then trigger Flow 2 for any approved FAQs.

## Phase 3: Flow 2 — FAQ Document Generation & Publish

**Trigger: **Called as a child flow from Flow 1 (recommended), or triggered by Source Document status changing to “Approved”.

### Step 3.1: Query Approved FAQs

- **List rows: **Query FAQ Items where SourceDocument = this document AND Status = Approved AND Active = Yes. Sort by Category then SortOrder.

### Step 3.2: Generate the FAQ Document

Two approaches — choose based on your preference:

**Option A: Word template (recommended for branded output):**

- Create a Word template with content controls in the Templates library.

- Use the Word Online connector’s “Populate a Microsoft Word template” action.

- The template should have a repeating section content control for FAQs.

- Map each FAQ’s question, answer, and source reference to the template fields.

**Option B: HTML to PDF:**

- Build an HTML string in a Compose action using the FAQ data.

- Use the Encodian connector or a custom HTTP action to convert HTML to PDF.

- This gives you full control over formatting without needing a template.

Regardless of approach, the output document should follow this structure for optimal retrieval:

[Agent Name] — Frequently Asked Questions

Source: [Document Name]

Last updated: [date]

Q: [Question text]

A: [Answer text]

Reference: [Source section] | [Link to original policy document]

───

Q: [Next question]

A: [Next answer]

Reference: [Source section] | [Link]

|  | **One FAQ Doc Per Source Document** Each source policy document generates its own FAQ document in the knowledge folder. This means the agent may have multiple FAQ documents (one per policy). This is intentional — it keeps content modular and makes it easy to update or remove a single policy’s FAQs without regenerating everything. |
| --- | --- |

### Step 3.3: Save to Knowledge Folder

- **Get the target folder: **Look up the agent’s KnowledgeFolderURL from the Agents table.

- **Create file: **Use SharePoint “Create file” action to save the generated document to the agent’s knowledge folder. File name: [Category]-[DocumentName]-FAQs.docx (or .pdf).

- **Update records: **Update the Source Document’s FAQDocURL with the SharePoint link. Update the Agent’s LastPublished timestamp.

## Phase 4: Flow 3 — Test Generation & Execution

**Trigger: **“When a file is created or modified” in the Knowledge library (with folder path filter for the specific agent’s subfolder), OR called as a child flow from Flow 2.

### Step 4.1: Generate Test Cases

- **Get the FAQ content: **If triggered by file creation, extract the text from the new FAQ document. If called as child flow, receive the FAQ JSON from Flow 2.

- **AI Builder GPT action: **Use the Test Case Generation prompt (see Section 4). Pass the FAQ content/JSON.

- **Parse and store: **Parse the test case JSON. For each test case, create or update a Test Case row in Dataverse: TestQuestion, ExpectedAnswer, LinkedFAQ (match by faq_id), Agent, LastResult = Not Run.

|  | **Test Case Updates** If this FAQ document has been regenerated (updated FAQs), the flow should update existing test cases rather than creating duplicates. Match on LinkedFAQ + VariationType to find existing test cases to update. |
| --- | --- |

### Step 4.2: Run the Test Suite

- **Create Test Run record: **RunName = AgentName + utcNow(), TotalTests = count, Trigger = Post-Publish.

- **Get Direct Line secret: **From the Agent record in Dataverse.

- **For each test case:**

- Start a new Direct Line conversation (POST /v3/directline/conversations).

- Send the test question (POST to /conversations/{id}/activities).

- Poll for response (GET activities with watermark, retry up to 10 times with 3-second delay).

- Evaluate response using AI Builder evaluation prompt (see Section 4).

- Update Test Case: ActualResponse, SimilarityScore, LastResult, LastRunDate.

- **Finalise: **Update Test Run with Passed/Failed/PassRate. Update Agent’s TestPassRate. Send notification card to the agent engineer in Teams.

|  | **Performance** 50 tests takes roughly 15–25 minutes via Direct Line. The flow runs asynchronously. Do not wait for it. The notification card is your signal to review results. |
| --- | --- |

## Phase 5: Copilot Studio Agent Setup

### Step 5.1: Create the Agent

- Go to copilotstudio.microsoft.com, select your environment.

- Create → New agent. Name it descriptively (e.g. “HR Policy FAQ Assistant”).

- Write a detailed description (used by generative orchestration for routing).

- Enable Generative orchestration (default).

### Step 5.2: Add SharePoint Knowledge Source

- On the Knowledge page, click Add knowledge.

- Select SharePoint.

- Browse to the agent’s subfolder under /Knowledge/ (e.g. /Knowledge/HR-Policy-Agent/).

- Give the knowledge source a descriptive name and detailed description.

- Wait for indexing to complete (status: Ready).

### Step 5.3: Configure Agent Settings

- **Turn off ungrounded responses: **Settings → Generative AI → toggle OFF “Allow the AI to use its own general knowledge.”

- **Set agent instructions: **Paste the agent instructions template (see Section 5).

- **Authentication: **Set to “Authenticate with Microsoft” for internal Teams agents.

### Step 5.4: Deploy to Teams

- Channels → Microsoft Teams → Turn on.

- Scope availability via Teams admin centre to the appropriate user groups.

### Step 5.5: Enable Direct Line (for Test Runner)

- Channels → Direct Line → Enable.

- Copy the secret and store it in the Agents table (DirectLineSecret column).

- Record the Bot ID from Settings → Agent details.

## Phase 6: BU Self-Service App (Canvas App)

A lightweight Canvas App that gives BU users control over their FAQ content without needing access to Copilot Studio or Dataverse directly.

### App Design

**Screen 1 — Home / Document List:**

- Shows all Source Documents where the current user’s security group matches the Agent’s BUSecurityGroup.

- Each document card shows: name, category, status, FAQ count, last updated.

- Buttons: “Upload New Document”, “View FAQs”, “Remove Document”.

**Screen 2 — Upload New Document:**

- Form with: file upload (attachment control), Category (dropdown), Target Agent (dropdown — filtered to agents the user has access to).

- On submit: creates a new item in the SharePoint Submissions library with the metadata, which triggers Flow 1.

- Confirmation message: “Document submitted. You’ll receive a Teams notification when FAQs are ready for review.”

**Screen 3 — FAQ List (for a selected document):**

- Gallery showing all FAQ Items linked to the selected Source Document.

- Each FAQ shows: question, answer (truncated), status, last approved date.

- Tap to edit: opens an edit form for that FAQ (question, answer, active toggle).

- Delete button per FAQ (sets Status = Archived, Active = No).

- Bulk action: “Regenerate FAQ Document” button that triggers Flow 2 to rebuild the knowledge file from current active FAQs.

### Security Scoping

The app uses the current user’s Entra group membership to filter visible content:

- On app load, call the Microsoft Graph connector (or Office 365 Groups connector) to get the user’s group memberships.

- Query the Agents table to find agents where BUSecurityGroup is in the user’s group list.

- Filter all galleries (documents, FAQs) to only show content linked to those agents.

- This means a user in the “FAQ-HR-Contributors” group sees only HR agent content, while a user in both HR and IT groups sees both.

|  | **Single App, Whole Organisation** One Canvas App is shared with the entire organisation. Security scoping happens at runtime via group membership. Users who aren’t in any FAQ security group see an empty home screen with a message to contact their admin. |

## Phase 7: Agent Management App (Model-Driven)

For agent engineers only. Full visibility and control over all agents, content, and testing.

### Sitemap Structure

- **Dashboard: **Overview of all agents with pass rates, FAQ counts, agents needing attention.

- **Agents: **Full CRUD on agent records. Sub-grids for related documents, FAQs, test runs.

- **Source Documents: **All uploaded documents across all agents. Filter by status, category, agent.

- **FAQ Items: **All FAQs across all agents. Views: Pending Review, Approved, Review Due, Failed Tests.

- **Test Cases: **All test cases. Key view: Failed Tests (showing expected vs actual side by side).

- **Test Runs: **History of all test executions. Trend analysis via charts.

### Key Features

**Request BU Review: **A custom button on the Agent form that triggers a Power Automate flow. The flow queries FAQs due for review (NextReviewDate <= today) and sends an Adaptive Card to the BU security group in Teams. The card summarises which FAQs need re-validation and links to the BU Self-Service App.

**Run Test Suite: **A custom button on the Agent form that triggers the test runner flow (Flow 3’s test execution portion) for the selected agent. Results populate in the Test Runs sub-grid when complete.

**Schedule Tests: **A scheduled Power Automate flow that runs weekly, iterating through all Active agents and executing their test suites. Results feed into the dashboard trends.

### Dashboard Components

- Card: Total Active Agents

- Card: Total Approved FAQs

- Card: Average Pass Rate (across all agents)

- Card: Agents Below 80% Pass Rate (attention needed)

- Chart: Pass Rate Trend (line chart, Test Runs over time, per agent)

- Chart: FAQs by Status (pie chart)

- Chart: FAQs by Category (bar chart)

- View: Agents Needing Attention (pass rate < 80% or review overdue)

# AI Prompts

All prompts used in the solution. Each is designed for AI Builder’s “Create text with GPT using a prompt” action.

## Prompt 1: FAQ Extraction

**Used in: **Flow 1, Step 2.2

**Input variable: **{{DocumentContent}} — the extracted text of the policy document

You are a senior knowledge management specialist at a large

international law firm. Your task is to convert the ENTIRE content

of the following policy document into a comprehensive set of

Frequently Asked Questions (FAQs).

CRITICAL REQUIREMENTS:

1. COMPLETE COVERAGE: Every piece of substantive information in

   the document must be captured in at least one FAQ. Do not skip

   sections, paragraphs, tables, or details. If a section

   contains multiple distinct points, create separate FAQs for

   each one.

2. QUESTION STYLE: Write questions the way a real employee

   would ask them in a chat interface. Use natural, conversational

   language. Avoid bureaucratic or policy-reference phrasing.

   GOOD: "How many days of annual leave do I get?"

   BAD: "What is the annual leave entitlement per section 3.2?"

   GOOD: "What do I do if I need to work from home?"

   BAD: "What is the procedure for requesting remote work

   arrangements as outlined in the flexible working policy?"

3. ANSWER STYLE:

   - Lead with the direct answer, then add context if needed.

   - Include specific numbers, dates, thresholds, conditions.

   - If different rules apply to different groups (by role,

     tenure, region, grade), include ALL variations.

   - Each answer must be self-contained. A reader should not

     need to read other FAQs to understand this one.

   - Do NOT invent information not present in the document.

   - If the document is ambiguous on a point, note the

     ambiguity in the answer.

4. PROCESSES: For any process described (how to apply, request,

   report, escalate), create a FAQ with numbered step-by-step

   instructions in the answer.

5. EDGE CASES: For conditions, exceptions, and special

   circumstances, create dedicated FAQs.

   Examples: "What happens if I exceed my expense limit?"

   "Can I carry over unused leave to next year?"

6. TABLES AND SCHEDULES: If the document contains tables

   (e.g. pay grades, entitlement schedules, fee structures),

   convert each meaningful row or grouping into one or more

   FAQs. Do not try to reproduce tables in FAQ format.

7. OUTPUT FORMAT: Return ONLY a valid JSON object. No markdown

   formatting, no code fences, no explanatory text before or

   after the JSON.

{

  "source_document": "[document title or filename]",

  "extraction_date": "[today's date YYYY-MM-DD]",

  "category": "[HR|IT|Finance|Legal Ops|Facilities|Other]",

  "total_faqs": [number],

  "faqs": [

    {

      "id": 1,

      "question": "...",

      "answer": "...",

      "source_section": "[heading or page ref from document]"

    }

  ]

}

DOCUMENT CONTENT:

{{DocumentContent}}

## Prompt 2: Test Case Generation

**Used in: **Flow 3, Step 4.1

**Input variable: **{{FAQJson}} — the FAQ extraction JSON output

You are a QA specialist testing an AI FAQ agent at a large

international law firm. You have a set of FAQs the agent is

trained on. Generate TEST QUESTIONS phrased DIFFERENTLY from

the FAQ questions but answerable from the FAQ content.

REQUIREMENTS:

1. Generate 2–4 test questions per FAQ, up to a MAXIMUM of 100

   test questions total. If the FAQ set is large, prioritise:

   - Processes (how to do something)

   - Entitlements and thresholds (numbers, days, amounts)

   - Conditions, exceptions, and edge cases

   - Frequently confused or similar topics

2. VARIATION STRATEGIES — use a mix for each FAQ:

   - casual: "Hey, how do I sort out my expenses?"

   - scenario: "I need to book a flight for a client meeting

     next week, what's the process?"

   - partial: "What's the limit for expenses?"

   - negative: "Am I not allowed to book my own travel?"

   - followup: "And what if I go over the limit?"

3. Each test question MUST be paired with the EXPECTED ANSWER

   copied exactly from the corresponding FAQ.

4. OUTPUT: Return ONLY valid JSON. No markdown, no code fences.

{

  "test_cases": [

    {

# FAQ Agent Factory v2 — Technical Reference

## 1. Direct Line API Reference

### Start Conversation
```
POST https://directline.botframework.com/v3/directline/conversations
Authorization: Bearer {DirectLineSecret}
Content-Type: application/json
```
Response: `{ "conversationId": "abc123", "token": "...", "expires_in": 1800 }`

### Send Message
```
POST https://directline.botframework.com/v3/directline/conversations/{conversationId}/activities
Authorization: Bearer {DirectLineSecret}
Content-Type: application/json

{ "type": "message", "from": { "id": "test-runner" }, "text": "How many days of annual leave do I get?" }
```

### Poll for Response
```
GET https://directline.botframework.com/v3/directline/conversations/{conversationId}/activities?watermark={watermark}
Authorization: Bearer {DirectLineSecret}
```
Filter activities where `from.id` != "test-runner" to get bot response. Pass `watermark` from each response to next poll.

### Power Automate Implementation
1. HTTP POST → start conversation → get conversationId
2. HTTP POST → send test question
3. Initialize: watermark = "", botResponse = "", attempts = 0
4. Do Until: botResponse not empty OR attempts >= 10
   - HTTP GET activities with watermark
   - Apply to each: if from.id != "test-runner" AND type == "message" → set botResponse
   - Increment attempts
   - Delay 3 seconds
5. botResponse = agent's answer (or empty if timeout)

**Important:** Create a NEW conversation per test question to avoid context bleed.

## 4. Knowledge File Format

Generated FAQ documents must follow this format for optimal Copilot Studio retrieval:

```
[Agent Name] — Frequently Asked Questions
Source: [Original Policy Document Name]
Last updated: [date]

Q: [Question — natural, conversational language]
A: [Answer — direct, specific, self-contained]
Reference: [Section name] | [SharePoint link to original document]

---

Q: [Next question]
A: [Next answer]
Reference: [Section] | [Link]

---
```

### Format rules:
- Q:/A: prefixes create clear chunk boundaries
- Blank line between Q and A stays within the same chunk
- Separator (---) between FAQ pairs reinforces boundaries
- Each answer is self-contained (no cross-references)
- Source links give the agent grounding for citations
- No tables (chunk poorly), no nested bullets, no intro text

## 7. Troubleshooting

### AI Builder returns invalid JSON
**Fix:** Compose: `trim(replace(replace(output,'```json',''),'```',''))`

### Adaptive Card too large
**Fix:** Switch to summary card (show first 3-5 FAQs) + link to BU Self-Service App.

### Agent says "I don't know" for content that exists
**Causes:** Knowledge not indexed yet (wait 30 mins), Q/A split across chunks (check document format), auth mismatch.
**Fix:** Test in Copilot Studio test pane (shows knowledge source hits). Check file status in Knowledge page.

### Direct Line 403
**Fix:** Verify Direct Line channel enabled in Copilot Studio. Regenerate secret. Update Agents table.

### Test pass rate drops after content update
**Fix:** Check if old test cases reference removed FAQs. Clean up orphaned test cases. Re-run test generation.

### Canvas App shows no content for a user
**Fix:** Check user's Entra group membership matches the Agent's BUSecurityGroup. Verify the Office365Groups connector is added to the app.

### SharePoint connector auth errors in Copilot Studio
**Fix:** Ensure agent auth is set to "Authenticate with Microsoft". User needs read access to the SharePoint site. Check Sites.Read.All and Files.Read.All permissions.

3.1  INTRODUCTION
----------------------------------------------------------------

3.1.1  Purpose

FAQ Forge is a repeatable system for creating, managing, and 
testing Microsoft Copilot Studio FAQ agents at [Firm Name]. It 
enables business units to self-serve the creation and 
maintenance of FAQ content from their policy documents while 
giving the agent engineering team centralised oversight, 
quality assurance, and lifecycle management.

3.1.2  Background and Drivers

The firm currently builds Copilot Studio agents on a per-project 
basis, with significant manual effort required to:

  - Convert policy documents into FAQ-ready content
  - Coordinate review and approval cycles with business units
  - Validate agent answer quality on an ongoing basis
  - Maintain content currency as policies evolve

This results in inconsistent agent quality, slow time-to-deploy, 
heavy dependency on the engineering team for routine content 
updates, and limited visibility into agent performance over time.

FAQ Forge addresses these issues by:

  - Automating FAQ extraction from policy documents using AI
  - Providing a structured BU approval workflow via Microsoft Teams
  - Auto-generating knowledge source files from approved content
  - Running automated regression tests after every content change
  - Giving business units self-service tools for ongoing maintenance

3.1.3  Scope

In Scope:
  - All internal-facing Copilot Studio FAQ agents at the firm
  - Agents deployed via Microsoft Teams
  - Policy and procedure content from any business unit
  - Automated test coverage and regression detection
  - Content lifecycle management (create, approve, edit, retire)

Out of Scope:
  - Client-facing or external agents
  - Agents requiring complex multi-step workflows (e.g. transactional 
    agents that perform actions in line-of-business systems)
  - Voice-enabled agents
  - Multi-language content (initial release: English only)

3.1.4  Document Conventions

  - "FAQ" refers to a single question-and-answer pair
  - "Source Document" refers to an original policy document 
    submitted by a business unit
  - "Knowledge Source" refers to a SharePoint folder configured 
    in Copilot Studio as a grounding data source
  - "Agent" refers to a Copilot Studio bot deployed to Teams

3.4  SOLUTION OVERVIEW
----------------------------------------------------------------

3.4.1  Conceptual Architecture

FAQ Forge consists of three Power Automate flows, two Power Apps, 
five Dataverse tables, and a SharePoint structure that serves as 
the knowledge source for Copilot Studio agents.

The end-to-end user journey:

  1. SUBMIT
     A BU user uploads a policy document via SharePoint with 
     metadata (category, target agent, approver).

  2. EXTRACT
     A flow extracts the document text using AI Builder, then 
     uses a GPT prompt to convert the entire document into 
     structured FAQ pairs.

  3. REVIEW
     The submitter receives an Adaptive Card in Teams. They can 
     approve all, edit individual FAQs inline, or open the 
     self-service app for detailed review.

  4. PUBLISH
     On approval, a flow generates a formatted FAQ document from 
     a Word template and saves it to the agent's SharePoint 
     knowledge folder. Copilot Studio detects the new file and 
     re-indexes automatically.

  5. TEST
     A second flow generates paraphrased test questions for each 
     FAQ, runs them against the agent via the Direct Line API, 
     evaluates responses using AI Builder, and stores results.

  6. SELF-SERVE
     BU users manage their own content via the FAQ Forge 
     Self-Service Canvas App, scoped by security group.

  7. MONITOR
     Agent engineers oversee everything via the FAQ Forge 
     Management App (model-driven), with dashboards, test trends, 
     and the ability to request BU reviews on demand.

4.5.1  AI Components

  Component 1: FAQ Extraction
    Technology: AI Builder "Create text with GPT" action
    Model: GPT-4o or GPT-4.1 (per AI Builder default)
    Purpose: Convert policy document text into structured FAQ pairs
    Input: Plain text extracted from PDF/DOCX
    Output: JSON array of FAQ objects

  Component 2: Test Case Generation
    Technology: AI Builder "Create text with GPT" action
    Model: GPT-4o or GPT-4.1
    Purpose: Generate paraphrased test questions from FAQs
    Input: FAQ JSON array
    Output: JSON array of test cases (max 100)

  Component 3: Test Response Evaluation
    Technology: AI Builder "Create text with GPT" action
    Model: GPT-4o or GPT-4.1
    Purpose: Compare expected vs actual answer, return similarity score
    Input: Two text strings
    Output: JSON with score (0-1) and pass/fail boolean

  Component 4: FAQ Agent Response Generation
    Technology: Copilot Studio (Azure OpenAI under the hood)
    Model: Default (currently GPT-4o or GPT-4.1)
    Purpose: Generate user-facing responses grounded in knowledge sources
    Configuration: Generative orchestration enabled, 
                   "Allow ungrounded responses" disabled

    - Scheduled weekly run of all agents' test suites
    - Pass rate threshold: 80%

  - All NFRs measured and within thresholds

  - Information Governance has signed off

4.10  PERFORMANCE AND SCALABILITY
----------------------------------------------------------------

4.10.1  Expected Volumes (Year 1)

  - Number of agents: 5-15
  - FAQs per agent: 50-300
  - Total FAQs across system: up to 5,000
  - Source documents submitted per month: up to 50
  - Test cases per agent: 100-500
  - Test runs per week per agent: 1-2
  - End-user agent interactions per day: estimate to be 
    determined per BU

4.10.2  Performance Considerations

  - FAQ extraction takes 1-5 minutes per document depending on 
    size
  - Test suite execution takes 15-25 minutes per 50 tests
  - SharePoint re-indexing after file save: 5-30 minutes
  - Adaptive Card response within seconds

4.10.3  Scaling Strategy

  - Vertical: Power Platform handles scaling automatically for 
    most components
  - Dataverse: storage scales linearly with content volume
  - AI Builder: monitored monthly; credit packs added as needed
  - Direct Line: rate-limited to 1 message per second per 
    conversation; test runner respects this

4.10.4  Bottlenecks Identified

  - SharePoint re-indexing delay is the main user-visible latency
  - AI Builder GPT response time is variable (10-60 seconds)
  - Direct Line polling adds overhead to test execution

4.13  GLOSSARY
----------------------------------------------------------------

  Adaptive Card
    A JSON-based UI element rendered in Microsoft Teams (and 
    other hosts) used here for FAQ review workflows.

  Agent
    A Copilot Studio bot deployed to Microsoft Teams that 
    answers user questions using configured knowledge sources.

  AI Builder
    Microsoft's low-code AI service within Power Platform, used 
    here for GPT-based prompts.

  Canvas App
    A Power Apps application built with a free-form designer, 
    used here for the BU Self-Service interface.

  Copilot Studio
    Microsoft's platform for building and deploying AI agents.

  Dataverse
    Microsoft's structured data platform used for managing FAQ 
    content, agents, and test results.

  Direct Line API
    The REST API for sending messages to and receiving responses 
    from a Bot Framework or Copilot Studio agent. Used here for 
    automated testing.

  FAQ
    A Frequently Asked Question entry consisting of a question 
    and its approved answer.

  File Group
    A Copilot Studio knowledge source feature that groups files 
    with shared instructions. Not used in this solution (we use 
    SharePoint folders instead).

  Generative Orchestration
    Copilot Studio's mode where the AI orchestrator decides 
    which knowledge sources, tools, or topics to invoke based 
    on the user's message.

  Knowledge Source
    A data source configured in Copilot Studio that the agent 
    uses to ground its responses. In FAQ Forge, this is a 
    SharePoint folder per agent.

  Model-Driven App
    A Power Apps application generated from Dataverse table 
    structures, used here for the engineering Management App.

  Power Automate
    Microsoft's workflow automation service used for all 
    backend orchestration.

  Power Fx
    The formula language used in Power Apps and Copilot Studio.

  Source Document
    An original policy document submitted by a BU for FAQ 
    extraction.

  Test Case
    A paraphrased version of a FAQ question used to validate 
    the agent can handle real-world phrasings.

  Test Run
    An execution of all test cases against an agent, producing 
    pass/fail results.
